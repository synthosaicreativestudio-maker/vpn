# VPN Project Rules

## Деплой VPN-инфраструктуры

При деплое кода панели на сервер:
1. Панель (`vpn-panel`) деплоится ТОЛЬКО на `37.1.212.51` (SCP → restart)
2. VPN-сервер `38.180.81.181` — только Xray, Caddy-прокси, SSH-туннель
3. Релей `185.4.67.223` — только Xray bridge + dokodemo-door
4. После деплоя панели — ВСЕГДА проверять: `curl http://37.1.212.51:8085/health`
5. После изменения кода — ОБЯЗАТЕЛЬНО обновить код на сервере:
   ```
   scp panel/*.py root@37.1.212.51:/root/vpn/panel/
   ssh root@37.1.212.51 "systemctl restart vpn-panel vpn-bot"
   ```
6. НИКОГДА не запускать vpn-panel на VPN-сервере 38.180.81.181
7. Health-check WARP: использовать ТОЛЬКО `cloudflare.com/cdn-cgi/trace` (проверять HTTP 200 + `warp=on`).
   НИКОГДА не использовать `google.com` — Google банит shared WARP IP (302→captcha).
   ChatGPT/OpenAI тоже блокируют WARP IP (403). Не подходят для мониторинга.
8. При ложных алертах мониторинга — СНАЧАЛА проверить, реально ли сервис сбоит:
   вручную протестировать через `warp-cli status` и `curl -x socks5://127.0.0.1:40000`.
   Не доверять слепо автоматическим алертам.
9. **Blue-Green Deployment (КРИТИЧЕСКИ ВАЖНО):**
   Порядок любых доработок VPN-инфраструктуры:
   a) **Шаг 1:** Переключить весь трафик на РЕЗЕРВНЫЙ канал (он становится основным).
   b) **Шаг 2:** Все доработки делать ТОЛЬКО на освободившемся канале (бывший основной, теперь idle).
   c) **Шаг 3:** После завершения доработки — сообщить пользователю. Ждать его тестирование.
   d) **Шаг 4:** После подтверждения пользователя — переключить доработанный канал на основной, а бывший резервный оставить как запасной.
   **НИКОГДА не дорабатывать канал, через который сейчас идёт живой трафик.**
10. После обновления Xray на ЛЮБОМ сервере — ОБЯЗАТЕЛЬНО проверить:
    a) `grep serverName config.json` на соответствие SNI (relay outbounds → US inbounds)
    b) Запустить health-check: `systemctl start xray-failover.service` + проверить логи
    c) Перезапустить SSH-туннели на всех связанных серверах
11. URL в routing-профиле Happ — ТОЛЬКО HTTPS (iOS ATS блокирует HTTP).
    Geo-файлы: `https://sub.synthosai.ru:8086/sub/geo/...`
12. При установке failover-скриптов — ВСЕГДА проверять наличие systemd timer:
    `systemctl is-active xray-failover.timer`
13. **Параллельные ветки Blue-Green (КРИТИЧЕСКИ ВАЖНО):**
    - В подписке пользователя всегда 6 протоколов — 3 от Ветки 1 (Blue, основная) и 3 от Ветки 2 (Green, резервная).
    - Ветка 1 (порты 443, 2053): **ДОРАБАТЫВАЕТСЯ ТОЛЬКО когда Ветка 2 стабильно работает**.
    - Ветка 2 (порты 10443, 12053): используется для тестирования и доработок.
    - Порядок: доработки → только на Ветке 2 → тестирование пользователем → если стабильно, переносить изменения на Ветку 1.
    - **НИКОГДА не менять обе ветки одновременно.**
