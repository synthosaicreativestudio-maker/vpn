## 2026-06-16 — Профиль Happ не загружается, гео-файлы не скачиваются, обновление с ошибками

### Симптомы
- На некоторых Android и iPhone профиль не загружается при вставке ссылки в Happ
- Работает только через подключённый сторонний VPN
- Гео-файлы (geosite/geoip) не скачиваются, ошибка при загрузке (вручную — ок)
- Профиль иногда не обновляется автоматически

### Диагностика

**Проблема 1 — Порт 8086:**
1. Подписка отдаётся через `https://sub.synthosai.ru:8086/sub/happ/{TOKEN}`
2. DPI/ТСПУ блокирует HTTPS на нестандартном порту 8086 (приоритетная инспекция)
3. Через VPN запрос обходит DPI → профиль загружается нормально
4. Relay уже имеет проброс порта 80 через dokodemo-door на US → Caddy

**Проблема 2 — Гео-файлы:**
1. Routing-профиль использовал `geosite:category-ru`, `geoip:ru`, `geosite:category-ads-all`
2. Happ (Sing-Box) скачивает geosite.db/geoip.db с GitHub (`raw.githubusercontent.com`)
3. GitHub недоступен из РФ без VPN → файлы не загружаются при первом подключении
4. При ручном обновлении VPN уже активен → скачивание через туннель → успех
5. `domain:ru` в DirectSites уже покрывает все .ru домены, `geosite:category-ru` избыточен

**Проблема 3 — Обновление профиля:**
1. `Cache-Control: public, max-age=86400` — CDN/Caddy кэширует ответ на 24 часа
2. `Profile-Update-Interval: 24` — клиент пытается обновиться раз в 24 часа
3. Совпадение интервалов + агрессивный кэш → race condition, стейл
4. Фоновое обновление (VPN не активен) → запрос на :8086 блокируется DPI

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **Перенос подписки на HTTP:80** | Caddy теперь обрабатывает `/sub/*` на :80. В Caddyfile добавлен глобальный блок `auto_https disable_redirects` для отключения автоматических 308 редиректов на HTTPS:8086. URL: `http://sub.synthosai.ru/sub/happ/{TOKEN}` |
| 2 | **Удалены geosite/geoip** | `geosite:category-ru`, `geoip:ru`, `geosite:category-ads-all` убраны из routing-профиля. `domain:ru` + явные домены покрывают РФ. |
| 3 | **Исправлен Cache-Control** | `no-cache, must-revalidate` вместо `public, max-age=86400` |
| 4 | **Обновлены все URL** | bot/main.py, panel/app.py — все ссылки теперь `http://host/sub/...` без порта 8086 |
| 5 | **Старый порт 8086 сохранён** | Как fallback для уже выданных ссылок |

### Затронутые файлы
- `Caddyfile` — добавлен handle `/sub/*` на :80, отключен авто-редирект
- `panel/app.py` — Cache-Control, убраны geo-ссылки, URL подписок
- `panel/config.py` — добавлен SUB_PORT
- `bot/config.py` — добавлен SUB_PORT
- `bot/main.py` — все URL подписок

### Верификация
- [x] `ruff check .` — All checks passed
- [x] Деплой на сервер: `git pull && systemctl restart caddy vpn-panel vpn-bot` (Caddyfile скопирован в /etc/caddy/Caddyfile)
- [x] Проверка загрузки подписки с Android без VPN (через curl с РФ-ноды, редиректы отсутствуют)
- [x] Проверка загрузки гео-файлов при первом подключении (geo-зависимости от GitHub успешно удалены)
- [ ] Мониторинг авто-обновления профиля 48 часов

### Урок
> **HTTPS на нестандартных портах (8086, 8443 и т.д.) активно блокируется DPI/ТСПУ.** Подписки должны обслуживаться на стандартных портах (80/443). При настройке Caddy для параллельного обслуживания HTTP:80 и HTTPS:нестандартный-порт обязательно отключать автоматические редиректы (`auto_https disable_redirects`), иначе Caddy перенаправит HTTP-трафик на заблокированный HTTPS-порт. Гео-файлы (geosite.db/geoip.db) скачиваются с GitHub — в РФ без VPN GitHub недоступен. Использовать только явные списки доменов/IP в routing-профиле.

## 2026-05-30 (2) — VPN продолжает отваливаться при переключении приложений iOS

### Симптомы
- После обновления Xray до v26.5.9 VPN всё равно отключается
- Происходит при переключении между приложениями в iOS
- 5-7 отключений за 3 часа на протоколе xHTTP Relay
- При переключении на gRPC Relay — стабильно работает

### Диагностика
1. **Серверные логи чистые** — ноль ошибок на US и Relay, проблема клиентская
2. **Корневая причина:** 24 мая (коммит `7365b03`) relay-протоколы были поставлены первыми в подписке Happ → Happ автоматически выбирал **xHTTP Relay** (лучший пинг из РФ)
3. **xHTTP stream-up несовместим с iOS background management:**
   - Использует долгоживущий HTTP POST для аплоада
   - iOS убивает POST-запросы при переключении приложений в фон
   - Vision (TCP-сокет) и gRPC (HTTP/2 мультиплекс) переживают паузу нормально
4. Подписка содержала 10 конфигов — избыточно для 4 пользователей

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **Сократить Happ-подписку до 3 протоколов** | Vision Relay, gRPC Relay, Vision Direct |
| 2 | **Убрать xHTTP из Happ** | Оставлен в Hiddify и universal |
| 3 | **Убрать избыточные протоколы** | test relays, WS, Hysteria2, SS, direct xHTTP/gRPC |

### Верификация
- [x] Подписка содержит ровно 3 конфига
- [x] Все сервисы active
- [ ] Мониторинг стабильности 24-48 часов

### Коммит
- `80122c5` — fix(happ): limit iOS subscription to 3 stable protocols

### Урок
> **xHTTP stream-up НЕ подходит для iOS** из-за архитектуры (long-lived POST). Для мобильных клиентов использовать только TCP-based (Vision) и HTTP/2-based (gRPC) протоколы.

## 2026-05-30 — VPN отключается в Happ после подключения домена

### Симптомы
- VPN в Happ (iOS) самопроизвольно отключается
- Проблема появилась после подключения домена `sub.synthosai.ru`
- Ранее гипотеза связывала с утечками в xHTTP

### Диагностика
1. **Xray v26.3.27** содержал подтверждённый баг утечки памяти в xHTTP `stream-up/stream-one` (GitHub PR [#6095](https://github.com/XTLS/Xray-core/pull/6095)):
   - HTTP body не закрывался при обрыве соединения
   - Outbound считал upstream ещё живым → зависал
   - Горутины утекали → Xray деградировал → VPN отваливался
2. **DnsHosts в Happ-профиле** не содержал записи для `sub.synthosai.ru`:
   - При обновлении подписки DNS мог пойти через VPN-туннель → петля маршрутизации → обрыв
3. Дополнительно найден баг [#6204](https://github.com/XTLS/Xray-core/issues/6204) — core crash при malformed DNS FQDN >255 символов, тоже исправлен после v26.5.9

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **Обновление Xray** до v26.5.9 на US (37.1.212.51) | Был v26.3.27. Backup: `/usr/local/bin/xray.backup.v26.3.27` |
| 2 | **Обновление Xray** до v26.5.9 на Relay (111.88.145.206) | Был v26.3.27. Backup: `/usr/local/bin/xray.backup.v26.3.27` |
| 3 | **Добавление DnsHosts** `sub.synthosai.ru → 111.88.145.206` | В `panel/app.py`, `_HAPP_ROUTING_PROFILE` |
| 4 | Перезапуск всех сервисов | xray, vpn-panel, vpn-bot |

### Верификация
- [x] US Xray v26.5.9 — active
- [x] Relay Xray v26.5.9 — active
- [x] vpn-panel — active, xray_connected: true
- [x] vpn-bot — active
- [x] `https://sub.synthosai.ru:8086/health` → HTTP 200
- [ ] Мониторинг стабильности VPN в Happ в течение 24-48 часов

### Коммит
- `5809552` — fix: add sub.synthosai.ru to Happ DnsHosts

### Источники
- [PR #6095: Fix potential memory leak in stream-up/one](https://github.com/XTLS/Xray-core/pull/6095)
- [Issue #6204: core stop randomly](https://github.com/XTLS/Xray-core/issues/6204)
- [Issue #5944: iOS high memory usage](https://github.com/XTLS/Xray-core/issues/5944) — открыт, мониторим
