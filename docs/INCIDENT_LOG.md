## 2026-07-16 (3) — Geo-базы не грузятся автоматом + failover timer не установлен

- Blue-Green failover timer не установлен на US-сервере (`xray-failover.timer` отсутствует).

### Диагностика (Loop Engineering)
1. **Geo-файлы:** GET запросы по `http://sub.synthosai.ru/sub/geo/geoip.dat` → HTTP 200 ✅. Но HTTPS (`https://sub.synthosai.ru/...`) → SSL error: `no alternative certificate subject name matches` ❌ (на relay нет Caddy, SSL-сертификат Reality = ozon.ru).
2. **Корневая причина:** iOS App Transport Security (ATS) блокирует HTTP-запросы по умолчанию → Happ не может скачать geo-файлы по HTTP URL.
3. **Решение:** Маршрут `sub.synthosai.ru:8086` → relay:8086 (dokodemo-door) → panel Caddy:8086 (HTTPS с валидным LE-сертификатом) → FastAPI. URL изменены на `https://sub.synthosai.ru:8086/...`.
4. **Failover:** Скрипт `xray-failover.sh` существовал на сервере, но systemd timer для его периодического запуска не был создан.

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **HTTPS geo URLs** | В `_HAPP_ROUTING_PROFILE` изменены `Geoipurl` и `Geositeurl` с `http://` на `https://sub.synthosai.ru:8086/...` |
| 2 | **Failover timer** | Созданы `xray-failover.service` и `xray-failover.timer` на US-сервере. Timer запускает health-check каждые 5 минут. |

### Верификация

## 2026-07-16 (2) — Пинг n/a на всех профилях (SNI mismatch после обновления Xray на релее)

### Симптомы
- В Happ все профили показывают `n/a` (пинг не проходит)
- VPN не подключается ни через Relay, ни через Direct

### Диагностика (Loop Engineering: Discover→Plan→Execute→Verify)
1. **Xray US и Relay активны**, все порты слушают — порты не причина.
2. **TLS handshake тест:** `openssl s_client -connect 185.4.67.223:443 -servername ozon.ru` → сертификат Ozon ✅ (inbound relays работают)

4. **Корневая причина:** При обновлении Xray на релее (26.3.27 → 26.5.9) мы перезапустили сервис, но не проверили соответствие SNI outbounds ↔ inbounds US-сервера.

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **Замена SNI в outbounds релея** | `sed -i 's/"serverName": "yandex.ru"/"serverName": "dzen.ru"/g'` на `/usr/local/etc/xray/config.json` |
| 2 | **Перезапуск Xray на релее** | `systemctl restart xray` → active |
| 3 | **Обновление .stable** | `cp config.json config.json.stable` — чтобы автооткат не вернул сломанный SNI |
| 4 | **Перезапуск туннелей** | `vpn-relay-tunnel` на панельном сервере |

### Верификация

## 2026-07-15 — Ложные алерты WARP каждую минуту (Google банит shared WARP IP)

### Симптомы
- Бот мониторинга шлёт алерты каждую минуту: `🚨 [VPN-US] Сбой WARP! Попытка автоматического восстановления не удалась. Тест Google вернул код: 302.`
- Код чередуется между 302 и 000 (таймаут при частых перезапусках warp-svc)
- Пользователи при этом жалоб не предъявляли

### Диагностика
1. **WARP работает исправно:** `warp-cli status → Connected`, `warp=on` в trace, SOCKS5 порт 40000 слушает
2. **Все западные сервисы через WARP доступны:** Instagram 200, YouTube 200, GitHub 200, Discord 200, Spotify 200, Netflix 200
3. **Google банит WARP IP:** `curl -x socks5://127.0.0.1:40000 https://www.google.com` → 302 → `/sorry/index` (CAPTCHA). С `-L` → 429 (rate limit). IP выхода: `104.28.195.105` (shared Cloudflare, colo: ORD)

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **Заменён тест-URL** | `google.com` → `cloudflare.com/cdn-cgi/trace` в `xray-failover.sh` |
| 2 | **Добавлена проверка warp=on** | Скрипт теперь проверяет не только HTTP-код (200), но и наличие `warp=on` в ответе Cloudflare trace |
| 3 | **Деплой на сервер** | SCP → `38.180.81.181:/usr/local/bin/xray-failover.sh` |
| 4 | **Тестовый запуск** | `Xray health check passed successfully.` — ложных алертов больше нет |

## 2026-07-08 — Нестабильность и падение Cloudflare WARP на VPN-сервере (wgcf keys expired)

### Симптомы
- У пользователей перестали открываться Google, Instagram и другие западные ресурсы.
- В логах Xray: `taking detour [WARP]` выполняется, но данные не идут.
- Ручная диагностика на VPN-сервере (`ping -c 3 -W 5 -I 172.16.0.2 1.1.1.1`) показала 100% packet loss.
- Проблема аналогична инциденту от 07.07 (ключи WireGuard/wgcf протухли менее чем за сутки).

### Диагностика
1. **Ненадежность wgcf:** Неофициальный клиент WireGuard генерирует временные ключи, которые Cloudflare часто аннулирует или блокирует без предупреждения. Требуется ручное переподключение и обновление `secretKey` в конфиге Xray каждые 1-2 дня.
2. **Альтернативное решение:** Установка официального демона `cloudflare-warp` (`warp-cli`), который умеет работать в режиме локального SOCKS5-прокси, автоматически управляет жизненным циклом ключей и поддерживает стабильное соединение.

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **Установка warp-cli** | Установлен официальный клиент Cloudflare на сервер `38.180.81.181` |
| 2 | **Настройка WARP в режиме прокси** | Зарегистрировано новое устройство, выставлен режим SOCKS5-прокси на порту 40000 (`warp-cli mode proxy`, `warp-cli proxy port 40000`) и запущено подключение |
| 3 | **Миграция Xray config** | Изменен аутбаунд `WARP` в `/etc/xray/config.json`: вместо протокола `wireguard` настроен протокол `socks` на `127.0.0.1:40000` |
| 4 | **Удаление wg0** | Локальный интерфейс WireGuard `wg0` удален, служба WireGuard больше не используется |
| 5 | **Обновление failover-скрипта** | В `scripts/xray-failover.sh` добавлена автопроверка WARP через curl с автоматическим рестартом `warp-svc` в случае сбоя |
| 6 | **Обновление stable-конфига** | Обновлен `/etc/xray/config.json.stable` на VPN-сервере |
| 7 | **Перезапуск служб** | Перезапущена служба `xray` на VPN-сервере и панель `vpn-panel` на панельном сервере `37.1.212.51` |

### Верификация
- [x] Статус `warp-cli status` → `Connected`
- [x] Запрос `curl -x socks5://127.0.0.1:40000 https://www.google.com` возвращает код 200
- [x] Логи Xray подтверждают успешную маршрутизацию: `taking detour [WARP] for ...`
- [x] Скрипт `xray-failover.sh` успешно проходит тест работоспособности локально и на сервере
- [x] Панель управления доступна, статус здоровый

### Урок
> **Использование неофициальных WireGuard-генераторов (wgcf) для Cloudflare WARP в production-окружениях приводит к частым отвалам ключей.** Официальный клиент `cloudflare-warp` в режиме SOCKS5 прокси — надежное, долговечное решение, которое само следит за регистрацией и ключами.

## 2026-06-16 — Профиль Happ не загружается, гео-файлы не скачиваются, обновление с ошибками

### Симптомы
- На некоторых Android и iPhone профиль не загружается при вставке ссылки в Happ
- Работает только через подключённый сторонний VPN
- Гео-файлы (geosite/geoip) не скачиваются, ошибка при загрузке (вручную — ок)
- Профиль иногда не обновляется автоматически

### Диагностика

**Проблема 1 — Порт 8086:**
1. Подписка отдаётся через `https://sub.synthosai.ru:8086/sub/happ/{TOKEN}`
2. DPI/ТСПУ блокирует HTTPS на нестандартном порту 8086 (приоритетная инспекция)
3. Через VPN запрос обходит DPI → профиль загружается нормально
4. Relay уже имеет проброс порта 80 через dokodemo-door на US → Caddy

**Проблема 2 — Гео-файлы:**
1. Routing-профиль использовал `geosite:category-ru`, `geoip:ru`, `geosite:category-ads-all`
2. Happ (Sing-Box) скачивает geosite.db/geoip.db с GitHub (`raw.githubusercontent.com`)
3. GitHub недоступен из РФ без VPN → файлы не загружаются при первом подключении
4. При ручном обновлении VPN уже активен → скачивание через туннель → успех
5. `domain:ru` в DirectSites уже покрывает все .ru домены, `geosite:category-ru` избыточен

**Проблема 3 — Обновление профиля:**
1. `Cache-Control: public, max-age=86400` — CDN/Caddy кэширует ответ на 24 часа
2. `Profile-Update-Interval: 24` — клиент пытается обновиться раз в 24 часа
3. Совпадение интервалов + агрессивный кэш → race condition, стейл
4. Фоновое обновление (VPN не активен) → запрос на :8086 блокируется DPI

### Решение
| # | Действие | Детали |
|---|----------|--------|
| 1 | **Перенос подписки на HTTP:80** | Caddy теперь обрабатывает `/sub/*` на :80. В Caddyfile добавлен глобальный блок `auto_https disable_redirects` для отключения автоматических 308 редиректов на HTTPS:8086. URL: `http://sub.synthosai.ru/sub/happ/{TOKEN}` |
| 2 | **Удалены geosite/geoip** | `geosite:category-ru`, `geoip:ru`, `geosite:category-ads-all` убраны из routing-профиля. `domain:ru` + явные домены покрывают РФ. |
| 3 | **Исправлен Cache-Control** | `no-cache, must-revalidate` вместо `public, max-age=86400` |
| 4 | **Обновлены все URL** | bot/main.py, panel/app.py — все ссылки теперь `http://host/sub/...` без порта 8086 |
| 5 | **Старый порт 8086 сохранён** | Как fallback для уже выданных ссылок |

### Затронутые файлы
- `Caddyfile` — добавлен handle `/sub/*` на :80, отключен авто-редирект
- `panel/app.py` — Cache-Control, убраны geo-ссылки, URL подписок
- `panel/config.py` — добавлен SUB_PORT
- `bot/config.py` — добавлен SUB_PORT
- `bot/main.py` — все URL подписок

### Верификация
- [x] `ruff check .` — All checks passed
- [x] Деплой на сервер: `git pull && systemctl restart caddy vpn-panel vpn-bot` (Caddyfile скопирован в /etc/caddy/Caddyfile)
- [x] Проверка загрузки подписки с Android без VPN (через curl с РФ-ноды, редиректы отсутствуют)
- [x] Проверка загрузки гео-файлов при первом подключении (geo-зависимости от GitHub успешно удалены)
- [ ] Мониторинг авто-обновления профиля 48 часов

### Урок
> **HTTPS на нестандартных портах (8086, 8443 и т.д.) активно блокируется DPI/ТСПУ.** Подписки должны обслуживаться на стандартных портах (80/443). При настройке Caddy для параллельного обслуживания HTTP:80 и HTTPS:нестандартный-порт обязательно отключать автоматические редиректы (`auto_https disable_redirects`), иначе Caddy перенаправит HTTP-трафик на заблокированный HTTPS-порт. Гео-файлы (geosite.db/geoip.db) скачиваются с GitHub — в РФ без VPN GitHub недоступен. Использовать только явные списки доменов/IP в routing-профиле.
