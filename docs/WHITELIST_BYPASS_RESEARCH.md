## 1. Как работает фильтрация

```
Пакет → [L3: IP в белом списке?]
         НЕТ → DROP (пакет исчезает)
         ДА  → [L7: SNI в чёрном списке?]
                ДА → RST (соединение рвётся)
                НЕТ → PASS
```

- **63 126 IP** в белом списке из 46 млн российских (0.14%)
- TCP 80/443/22 работают на whitelisted IP
- **UDP блокируется полностью** (QUIC, DNS, WireGuard, Hysteria2)
- Затрагивает **мобильный интернет** (домашние провайдеры пока нет)
- Работает неравномерно по регионам и провайдерам

## 5. Cloudflare — полный анализ

### 5.1 IP Cloudflare (проверено 11.05.2026)

| Сервис | IP | Подсеть | В белом списке? |
|--------|-----|---------|----------------|
| CDN | `104.16.x.x` | `104.16.0.0/12` | ❌ **НЕТ** — зарубежная ASN |
| Workers | `8.47.69.0` | Cloudflare | ❌ **НЕТ** |
| WARP | `162.159.192.1` | Cloudflare | ❌ **НЕТ** + UDP заблокирован |
| Tunnel | `198.41.192.x` | Cloudflare | ❌ **НЕТ** |

### 5.2 Вердикт по каждому сервису Cloudflare

#### ❌ Cloudflare WARP
- Использует **WireGuard (UDP)** — UDP полностью блокируется при БС
- IP не в белом списке
- В "обычном" режиме тоже нестабилен в РФ (блокируется ТСПУ)
- **Вердикт: БЕСПОЛЕЗЕН при белых списках**

#### ❌ Cloudflare Workers
- IP Workers (`8.x.x.x`) не в белом списке
- В некоторых регионах Workers уже заблокированы в РФ
- Даже если бы работали — это HTTP-прокси, не VPN-туннель
- **Вердикт: НЕ РАБОТАЕТ при белых списках**

#### ❌ Cloudflare CDN (VLESS+WS через CF)
Схема: `Клиент → CF CDN (104.16.x.x) → WS → наш VPS → Интернет`

- IP CDN (`104.16.0.0/12`) **не в белом списке** → DROP на L3
- РКН рекомендовал отказаться от CF, TLS ECH блокируется
- Hiddify и Happ поддерживают VLESS+WS+CDN, **но при БС CDN недоступен**
- **Вердикт: НЕ РАБОТАЕТ при белых списках**

#### ❌ Cloudflare Tunnel (Argo)
- IP тоннелей (`198.41.192.x`) не в белом списке
- Активно блокируется ТСПУ
- **Вердикт: НЕ РАБОТАЕТ при белых списках**

### 5.3 Единственный сценарий где CF полезен

**Cloudflare CDN работает НА ДОМАШНЕМ интернете** (без БС). Схема:

```
Домашний Wi-Fi → CF CDN → VLESS+WS → наш US VPS → Интернет
```

Это даёт скрытие IP нашего сервера за CDN. Полезно **не для обхода БС**, а для защиты от блокировки IP нашего US-сервера на домашних провайдерах.

## 12. Поиск открытых ресурсов в whitelisted подсетях

### 12.1 Суть идеи

В белом списке ~63 000 IP. На многих из них крутятся серверы с открытыми сервисами. Можно найти:
- **Открытые SSH** (порт 22 без ключа или со слабым паролем)
- **Открытые HTTP-прокси** (порт 8080/3128)
- **Открытые SOCKS5** (порт 1080)
- **Незакрытые API** (любой порт)
- **VPS с дефолтными настройками** (не настроен firewall)

## 13. Нестандартные методы обхода

### 13.1 Стеганография в легитимном трафике

**Идея:** Прятать VPN-данные внутри легитимного трафика к whitelisted сервисам.

```
┌─────────────┐     ┌──────────────────────┐     ┌──────────┐
│ 📱 Клиент   │────▶│ ozon.ru (whitelisted) │────▶│ Данные   │
│             │     │ Запросы с payload     │     │ скрыты в │
│ Стего-клиент│     │ в Cookie/Headers/Body │     │ HTTP     │
└─────────────┘     └──────────────────────┘     └──────────┘
```

**Каналы скрытия данных в HTTP/HTTPS:**

| Канал | Ёмкость | Заметность |
|-------|---------|-----------|
| Cookie заголовки | ~4 KB/запрос | Низкая |
| HTTP Headers (кастомные) | ~8 KB/запрос | Средняя |
| POST body (form data) | Неограничено | Средняя |
| URL параметры | ~2 KB/запрос | Низкая |
| Timing (задержки между запросами) | ~10 бит/сек | Очень низкая |
| Размер ответа (padding) | ~100 бит/запрос | Очень низкая |

**Проблема:** Очень низкая пропускная способность. Для мессенджеров — хватит, для видео — нет.

### 13.2 Abuse API российских сервисов

**Идея:** Использовать публичные API whitelisted сервисов как транспортный канал.

#### Telegram Bot API через whitelisted IP
- Telegram API серверы **могут быть** доступны через whitelisted подсети (проверить!)
- Bot API: `api.telegram.org` → отправляем данные как сообщения
- Задержка: ~100-500ms, пропускная способность: ~50 KB/s
- Двунаправленный канал: long polling / webhooks

#### Yandex Cloud Storage (S3-compatible)
- `storage.yandexcloud.net` — **в белом списке** (проверено)
- Загружаем/скачиваем файлы = канал данных
- Пропускная способность: до 10 MB/s
- Нужен аккаунт Yandex Cloud

- Отправляем данные через сообщения/стену/документы

### 13.4 DNS-over-HTTPS (DoH) через whitelisted резолверы

**Идея:** DNS-провайдер Яндекса (`dns.yandex.ru`) может быть в белом списке по HTTP/HTTPS. Используем DoH-запросы как транспорт.

```
Клиент → dns.yandex.ru:443 (HTTPS, DoH запрос)
       → Внутри: закодированные данные в DNS query name
       → DNS ответ содержит данные от нашего сервера
```

Отличие от обычного DNS-туннеля: **работает по TCP:443** (не UDP:53), через whitelisted IP.

## 14. Фундаментальные идеи — уровень физики и архитектуры связи

### 14.1 Почему белые списки вообще возможны

Белые списки работают потому что:
1. **Централизованная маршрутизация** — весь трафик идёт через узлы оператора
2. **Прозрачность L3** — IP назначения видно в каждом пакете
3. **Иерархическая топология** — телефон → базовая станция → ядро сети → интернет

**Фундаментальный вопрос:** Что если убрать хотя бы одно из этих условий?

### 14.2 Идея A: Mesh-сеть телефон-телефон

```
📱 Телефон A (нет интернета, БС)
    │ Wi-Fi Direct / Bluetooth
    ▼
📱 Телефон B (нет интернета, БС)
    │ Wi-Fi Direct
    ▼
📱 Телефон C (есть домашний Wi-Fi!)
    │ Wi-Fi → роутер → провайдер (без БС)
    ▼
🌐 Интернет
```

**Суть:** Если у одного человека в доме есть домашний интернет (без БС), он становится "выходной нодой" для соседей, у которых только мобильный (с БС).

| Плюс | Минус |
|------|-------|
| Не зависит от оператора | Дальность Wi-Fi Direct ~50м |
| Не проходит через ТСПУ | Нужен хотя бы 1 человек с Wi-Fi |
| Бесплатно | Сложно организовать |

**Технологии:** Apple AirDrop, Android Nearby Share, Wi-Fi Aware, Bluetooth Mesh

### 14.3 Идея B: Спутниковый интернет

| Сервис | Доступен в РФ? | При БС |
|--------|---------------|--------|
| **Starlink** | ❌ Нет (не работает официально) | ✅ Не зависит от ТСПУ |
| **OneWeb** | ⚠️ Частично (через Ростелеком) | ❌ Фильтруется |
| **Другие LEO** | ❌ | Теоретически обходит |

**Starlink** полностью обходит ТСПУ (сигнал идёт напрямую на спутник, минуя наземную инфраструктуру). Но нелегален и требует терминал.

### 14.4 Идея C: Радиомост (самая безумная)

```
📻 Передатчик (город с БС)
    │ Радиосигнал (UHF/VHF, LoRa, или Wi-Fi direction antenna)
    ▼
📻 Приёмник (место без БС, или с домашним интернетом)
    │ Ethernet → роутер → интернет
    ▼
🌐 Выход в сеть
```

**LoRa (Long Range):** До 15 км в городе, до 50 км на открытой местности. Скорость 0.3-50 Kbps. Хватит для мессенджеров.

**Направленная Wi-Fi антенна:** До 5-10 км. Скорость до 100 Mbps. Нужна прямая видимость.

### 14.5 Идея D: "Невидимый" канал — модуляция в легитимном трафике

**Фундаментальная идея:** ТСПУ проверяет **куда** идёт трафик (IP) и **что** в заголовке (SNI). Но не проверяет **паттерн** трафика.

```
Обычный HTTPS к ozon.ru:
  Запрос каждые 2 секунды, размер 1500 байт

Модулированный HTTPS к ozon.ru:
  Запрос каждые 2.001 / 2.003 / 2.002 секунды
  Разница в миллисекундах = закодированные данные
```

**Timing covert channel:** Кодируем информацию в **задержках между пакетами**:
- Задержка 1ms = бит "0"
- Задержка 3ms = бит "1"
- Скорость: ~500 бит/сек
- Детекция: практически невозможна

**Размерный covert channel:** Кодируем в **размере HTTP-ответов**:
- Подгружаем разные картинки с ozon.ru
- Выбор картинки = закодированное сообщение
- Нужен контроль над whitelisted сервером (или предсказуемый контент)

### 14.6 Идея E: Паразитирование на PKI/сертификатах

Из статьи (UPD автора): "есть темка с абузом сертификатов для обхода БС, но об этом позже"

**Гипотеза:** Let's Encrypt / Certificate Authority инфраструктура доступна через whitelisted IP. При выпуске сертификата происходит HTTP-01 challenge — это **легитимный входящий HTTP-запрос** на порт 80 вашего сервера. Возможно, можно использовать этот механизм как транспорт.

### 14.7 Идея F: "Интернет через почту" (Store-and-Forward)

Если нужен не realtime, а delayed access (например, проверить email, получить новости):

```
📱 Телефон → SMS → Сервер (по SMS gateway) → Интернет → Ответ → SMS → 📱
```

SMS работает **всегда** (это голосовая сеть, не IP). Через SMS-gateway можно отправлять URL, получать контент как серию SMS.

Пропускная способность: ~140 байт/SMS, ~1 SMS/сек = 140 байт/сек. Для текста — достаточно.
