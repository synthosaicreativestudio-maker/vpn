# 💳 Billing & Subscription Security

Этот skill описывает подходы и стандарты безопасности при генерации токенов подписок, ограничении несанкционированного доступа к панели и интеграции платежных шлюзов (например, эквайринга Т-Банка).

## 🔑 Генерация токенов и UUID

1. **Токены подписок (Subscription Tokens):**
   * Должны генерироваться криптографически стойким генератором случайных чисел (например, модуль `secrets` в Python).
   * Рекомендуемый формат: 32+ символов (буквы верхнего/нижнего регистра и цифры).
   * Пример генерации:
     ```python
     import secrets
     sub_token = secrets.token_urlsafe(24)  # ~32 символа
     ```

2. **UUID пользователей (UUID v4):**
   * Для протоколов VLESS/VMess всегда используйте стандартный UUID v4.
     ```python
     import uuid
     user_uuid = str(uuid.uuid4())
     ```

## 🔒 Защита API панели управления (FastAPI)

1. **API Key авторизация:**
   * Все административные endpoints должны быть защищены заголовком `X-API-Key` или авторизацией через Bearer токен.
   * Статический API ключ должен храниться в переменных окружения и сверяться в режиме константного времени `secrets.compare_digest` для предотвращения атак по времени (Timing Attacks).

2. **Ограничение частоты запросов (Rate Limiting):**
   * Эндпоинты подписок (`/sub/{token}`) запрашиваются клиентами автоматически раз в несколько часов. Для защиты от DDoS-атак и сканирования токенов необходимо настраивать лимиты на уровне веб-сервера (Caddy / Nginx) или приложения:
     ```text
     Caddy (caddy-ratelimit):
     rate_limit {
         zone sub_limit {
             key {path}
             events_per_period 10 1m
         }
     }
     ```

3. **Ограничение по IP (IP Limit enforcement):**
   * Панель должна контролировать количество уникальных IP-адресов, с которых скачивается/используется профиль за определенный промежуток времени.
   * Если лимит превышен (например, к одному токену подключено более 2-3 разных IP одновременно), подписка должна временно блокироваться.

## 💳 Интеграция с платежными системами (Эквайринг)

1. **Защита Webhook-уведомлений:**
   * Уведомления от банка об успешной оплате (Webhooks) должны проверяться на валидность подписи (подпись формируется банком на основе секретного ключа и параметров платежа).
   * Алгоритм формирования подписи банка должен в точности соблюдаться: сбор параметров, сортировка по алфавиту, конкатенация с паролем терминала и хеширование SHA-256.

2. **Безопасная обработка данных:**
   * Никогда не сохраняйте данные карт пользователей на стороне своей панели. Все платежные операции должны происходить на защищенных страницах платежного шлюза (Redirect/Frame интеграция).
   * Общайтесь с API банка только по протоколу HTTPS.
